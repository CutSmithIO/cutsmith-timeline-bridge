## Phase 2 — ir_diag.py key-collision bug fix

### Root cause

`build_summary()` was building `speed_by_time: dict[int, dict]` keyed by `timeline_start_us`, then using `c.timeline_start_us` to look up per-clip speed info for the inconsistency check.

When multiple tracks all start at `timeline_start_us = 0` (common for overlay / parallel tracks), the dict only retains the last entry written. With 0519's three tracks all starting at 0:

- The 0.5× clip's entry (`target_dur=72133333us`) overwrote both earlier entries.
- V1 (1.0×, `ir_duration=36066666us`) was looked up with the 0.5× hint → `ir_duration ≠ target_dur` → **false inconsistency**.
- V2 (2.0×, `ir_duration=18033333us`) got the same wrong hint → **false inconsistency with wrong speed label**.
- V3 (0.5×, `ir_duration=72133333us`) happened to match → no inconsistency, but for wrong reasons.

### Fix

Added `_SEG_ID_RE` to extract the segment UUID from `unsupported.detail` strings (`"segment XXXX has speed=…"`). Changed `speed_by_clip: dict[str, dict]` keyed by clip UUID. All lookups now use `c.clip_id.upper()`.

Same fix applied to `_clip_summary()` (controls the `speed` column in the markdown report).

### Files changed

- `scripts/ir_diag.py` — `_SEG_ID_RE` regex, `speed_by_clip` dict, two lookup sites

### Regression check

All 4 known samples pass with correct output after fix:
- 0509: `unsupported=0, inconsistencies=0`
- cutsmith: `unsupported=0, inconsistencies=0`
- cutsmith2: `unsupported=9 (speed ×1, speed_curve ×1, text/effects/…), inconsistencies=0`
- 0519: `unsupported=4 (speed ×2, speed_curve ×2), inconsistencies=0`

## Phase 4 — 0519V deep capability audit: key discoveries

### D1 — vol=0.001 is CapCut's soft-mute sentinel

CapCut sets `segment.volume = 0.001` (not 0.0) when a clip's embedded audio is "muted" but the clip is still active. Hard mute (main clip) is `volume=0.0`. Soft mute (B-roll overlays) is `0.001`.

**Impact on Smart Embedded Audio Fallback (v0.1.2 spec)**: the coverage threshold for "is this clip's audio effectively muted?" should be `volume ≤ 0.001`, not `volume == 0.0`.

### D2 — Filters live in `materials.effects` with `type=filter`

There is no separate `materials.filters[]` category. The `filter` track references materials in `materials.effects[]` where `mat["type"] == "filter"`. Example: "Film" (`type=filter`, `resource_id=6706773...`), "Vivid Verse" (`type=filter`).

**Impact on scan-assets classifier**: the `effects` category must be split by `type` field:
- `type=filter` → `AssetClass.CAPCUT_EFFECT` (LUT/filter sub-class in v0.4)
- `type=video_effect` → `AssetClass.CAPCUT_EFFECT`
- Others (transitions, etc.) → handled via `extra_material_refs` classifier

### D3 — Subtitle extraction requires two patterns

**Pattern A** (simple text, e.g. cutsmith2):
```
text_segment.material_id → materials.texts[].id
texts[].content (JSON) → .text
```

**Pattern B** (animated subtitle template, 0519V — type=`text_template_subtitle`):
```
text_segment.material_id → materials.text_templates[].id
  → text_templates[].text_info_resources[].text_material_id → materials.texts[].id
    → texts[].content (JSON) → .text
```

Pattern B is used when users apply CapCut's "AI Captions" or animated subtitle templates. The `non_text_info_resources[]` in the template can also embed sticker emojis alongside the text.

Subtitle text content is accessible in both patterns via `materials.texts[].content` → `json.loads()` → `.text`. `recognize_text` is the ASR fallback when content parsing fails.

Verified 0519V text content (8 cues, 29.5s–47.2s):
- `'god'`, `'you look so pretty when you tell me that you love me'`, `'hello nice to see you'`, etc.

### D4 — Cache asset classification from material `type` field

CapCut stores music library assets with `type=music` and SFX with `type=sound` in `materials.audios[]`. The original video's embedded audio is `type=video_original_sound`. These `type` values are sufficient to classify without inspecting the cache path.

Cache path (macOS): `~/Library/Containers/com.lemon.lvoverseas/Data/Movies/CapCut/User Data/Cache/music/...`

All three audio library items (Self Aware, Magic reveal, An Ideal Swoosh) have `source_platform=0` and `resource_id=""` — the resource_id field is unreliable for CapCut library tracks. The `type` field + cache path pattern is the reliable classifier.

### D5 — No speed changes in 0519V

All 11 video segments: `speed=1.0`, `source_dur == target_dur`. Clean. This sample does not exercise the speed path.

## Phase 6 — Revised v0.2 priority recommendation

Based on 0519V audit results:

### Priority order

| Priority | Feature | Reason |
|---|---|---|
| **P0 — Unblocked** | `export-srt` (subtitle extraction) | Pattern A + B both understood; text content accessible via `texts[].content.text` |
| **P0 — Unblocked** | `scan-assets` basic classification | `type` field + cache path sufficient for USER vs CAPCUT_MUSIC vs CAPCUT_SFX |
| **P1 — Gated** | `collect` (file copy + relink) | Gate: Premiere speed test result. If PR doesn't auto-apply implicit speed, XML rewrite for speed clips changes before collect can ship. |
| **P1 — Gated** | Explicit Time Remap `<filter>` | Gate: same Premiere speed test. Rises to P0 if implicit speed fails. |
| **P2** | Smart Embedded Audio Fallback | vol≤0.001 = soft mute sentinel confirmed. v0.1.2 spec still valid; implement after P0 lands. |
| **P3** | Classification refinement (sticker type, filter kind) | Pattern understood (D2, D4); defer to v0.4. |

### What 0519V confirmed is ready

- `export-srt`: green light. Both patterns understood. 8 subtitle cues with correct timing and text content verified.
- `scan-assets` core: green light. Audio `type` field + cache path sufficient for the most important classifications (user media vs CapCut library vs SFX).
- Embedded audio (A1 = `video_original_sound` → explicit track): already works correctly in v0.1.1.

### What 0519V surfaced as still open

- **Premiere speed test** (0519 sample): whether `end-start ≠ out-in` in FCP7 XML is auto-interpreted as constant speed. This is the biggest unknown in the current architecture and gates collect.
- **vol=0.001 mute threshold**: confirmed; needs a code update in the Smart Embedded Audio Fallback spec (`volume ≤ 0.001` instead of `volume == 0.0`).
- **Sequence resolution mismatch** (0519V is 1080×1920, 0519 is 1920×1080): no issues in the converter — resolution is read from `canvas_config` and passed through correctly in both cases.

## Decisions captured

1. **`scan-assets` and `export-srt` are the correct v0.2 targets.** Both patterns are understood. No structural unknowns.

2. **`collect` is gated on the Premiere speed test.** The test is a prerequisite for knowing whether collected XML needs Time Remap `<filter>` nodes inserted for speed clips.

3. **vol=0.001 is the soft-mute sentinel in CapCut.** The Smart Embedded Audio Fallback threshold updates from `volume == 0.0` to `volume ≤ 0.001`.

4. **Filters live in `materials.effects` (not a separate `filters` key).** Classification must split `effects[]` by the `type` subfield.

5. **Subtitle extraction uses `text_template.text_info_resources[].text_material_id` for Pattern B.** Direct `texts[]` lookup only works for Pattern A (plain text segments). Pattern B is used for CapCut's animated subtitle templates.

6. **ir_diag.py `speed_by_time` bug is fixed.** Do not rekey by `timeline_start_us` for any future per-clip lookups in ir_diag — use clip UUID.
