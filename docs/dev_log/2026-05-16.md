## Phase 1 — Modern Jianying 75 storage research

Tried `python3 -m cutsmith inspect …/JianyingPro/.../Timelines/<UUID>/draft_info.json`
and got `JSONDecodeError`. The file is a single ASCII line of base64-like
characters, ~7800 bytes. Sibling `template.tmp` is plaintext JSON but is
the empty project skeleton (`tracks: []`, `duration: 0`, etc.) — not the
real timeline.

### Decision: do NOT attempt decryption

Reasons captured in `docs/notes/modern_jianying_75_storage.md`:
- Compliance / ToS grey area
- Bytedance can rotate keys per app version; any solution is fragile
- Won't block the core value proposition, which is rough-cut migration
  on plaintext drafts

v0.1 scope explicitly narrowed to plaintext layouts. `detect` will refuse
encrypted drafts with `supported_status: unsupported_encrypted` and a
pointer to that notes file.

### Three layouts found in the wild

| Layout | Where draft lives | Found in |
|---|---|---|
| Legacy plaintext | `<project>/draft_content.json` | Old desktop / mobile builds |
| **Flat modern plaintext** | `<project>/draft_info.json` | CapCut Desktop 167.0.0 (sample `0509`) |
| Nested modern (often encrypted) | `<project>/Timelines/<UUID>/draft_info.json` | JianyingPro 75.0.0+, also CapCut Desktop 167.0.0 (samples `cutsmith` / `cutsmith2`) |

Initial `detect` missed the flat-modern layout; added a second pass after
seeing 0509. Precedence: nested > flat > legacy.

### Important non-obvious finding

CapCut Desktop 167.0.0 keeps `<project>/draft_info.json` and
`<project>/Timelines/<UUID>/draft_info.json` byte-identical (both
plaintext). The nested copy is the canonical one (matches the
`main_timeline_id` in `Timelines/project.json`); the flat copy is
probably a compatibility shim.

JianyingPro 75.0.0+ uses the nested layout AND encrypts it.

## Phase 3 — Sample 0509 smoke test → double-audio writer bug

0509 is a single-cut, 151.4s, 154-caption project. End-to-end conversion
succeeded structurally but produced `V1 + A1 + A2` where A1 and A2 both
referenced the same MP4 — duplicate audio at 2× level on Premiere import.

### Root cause

`writer/fcp7_xml.py:_build_audio_section` had a loop that emitted one
audio track per video track whenever `asset.has_audio=True`. In
practice CapCut always *also* creates a separate audio material entry
pointing at the same file (so the user can independently adjust audio
volume / mute), and that already lives in `Timeline.audio_tracks`. The
writer was double-counting.

### Fix

Writer is now a pure IR → XML transform. `Timeline.audio_tracks`
contains exactly the tracks to emit. `MediaAsset.has_audio` on video
assets is informational only (used by the report layer to flag
"silent video" cases when no explicit audio track covers the source).

### Files touched

- `cutsmith/writer/fcp7_xml.py` (delete auto-extract loop, add docstring point 7)
- `cutsmith/ir/__init__.py` (Timeline docstring re-states the contract)
- `cutsmith/report/compatibility.py` (drop the misleading "exported as
  separate audio tracks alongside the video" note; add an
  embedded-audio orphan detector)
- `cutsmith/__init__.py` + `README.md` wording updates
- `tests/test_pipeline.py` +4 regression tests pinning:
  - mock fixture → exactly 1 audio track
  - IR with no audio tracks → zero audio clipitems in XML
  - video clip with embedded audio not covered → report flag
  - video+audio assets sharing path → no flag

## Phase 4 — `cutsmith` multicut + P1 fixes

`cutsmith` sample: V1×7 butt-cut + V2 overlay + BGM + SFX, 60s. Reader
read cleanly. But the unsupported list had **14 noise items** before any
fix:

- 10 × `extra_refs` ("segment X has N extra material refs (transitions/filters/speed curves) — not exported") — every segment got one even when the refs were just default canvas / sound channel / 1.0× speed bookkeeping
- 4 × `speed` — every segment with `source_dur != target_dur` by 1µs got flagged as a speed change

### P1.1 — speed tolerance

CapCut writes µs-level rounding noise even on unedited clips:
`target=8833334us, source=8833333us`. Added `_SPEED_DUR_TOLERANCE_US = 1000`
(1ms ≈ 1/30 frame at 30fps) so sub-millisecond diffs are silent. Real
speed changes either set `segment.speed != 1.0` or diverge by >1ms.

### P1.2 — extra_material_refs categorisation

Added `_build_materials_index()` and `_classify_extra_ref()`. Each
`extra_material_refs` UUID is now dereferenced to its category:

- **Always real** (referenced ⇒ report under category label): transition, filter, effect, video_effect, audio_effect, audio_fade, mask, chroma_key, color_curve, hsl, video_stroke, …
- **Conditional** (predicate decides): speeds (speed != 1.0 or curve_speed), material_animations (animations array non-empty), vocal_separations (choice != 0), canvases (non-default color/image), sound_channel_mappings (non-default), material_colors (color clip or gradient), loudnesses (enable), beats (ai_beats or user_beats)
- **Always benign**: placeholder_infos
- **Unknown**: silently skipped (false negatives preferred to flooding the report)

Also removed material-scan for transitions/filters/effects/video_effects/audio_effects/masks in `_parse_assets` — they're now sourced from segment-level refs only. Trade-off: an unreferenced effect material won't be reported. That's intentional; unused materials have no on-timeline impact.

`tests/test_pipeline.py` +7 regression tests pinning the tolerance
boundaries and the classifier outcomes.

### Result on `cutsmith`

`unsupported items: 0` after P1 fixes. Clean baseline.

## Phase 5 — cutsmith2 stress test + 3 blockers

cutsmith2 covers captions + stickers + transitions + filters + effects +
**a real 0.640× variable-speed clip**. inspect showed 346 unknown_fields
(more than 0509's 266 — richer draft); IR diag reported a *real* speed
unsupported item plus 1 `ir_inconsistency` (the P3 latent bug — finally
exposed).

### Blocker 1 — P3: timeline_duration uses source_dur, not target_dur

The reader was setting `Clip.source_out_us = source_start + source_dur`,
and `Clip.timeline_duration_us` was a property derived as
`source_out - source_in`. So when CapCut had a 8.833s source slice
slowed into a 13.800s target slot, the IR said the clip occupied 8.833s
on the timeline — **downstream clips would drift**.

Fix:
- `Clip.timeline_duration_us` is now a real field (Optional[int], post_init derives from source range if not given)
- Reader sets `timeline_duration_us=target_dur` explicitly
- Writer's `end_frames` now comes from `clip.timeline_duration_us`, not from `out - in`

When `source_dur != target_dur`, the XML's `end-start ≠ out-in`. Premiere
interprets the ratio as implied speed. We do **not** emit an explicit
Time Remap `<filter>` — the user can override by setting Speed to 100%
in PR's clip menu.

This assumption (PR auto-interprets the duration mismatch) is the single
biggest unverified thing in v0.1.1. The creator validation checklist
calls it out as Sample 3's critical observation.

### Blocker 2 — embedded-audio warning was too loud

`cutsmith2` triggered `8 video clip(s) reference source files with
embedded audio that is not on any audio track…` styled as a `⚠` warning,
suggesting Premiere would import silent video. But the user had
deliberately used external BGM/SFX instead of original audio — common
workflow, not a defect.

Softened to a neutral note: "_Note: 8 video clip(s) contain embedded
audio that was not exported on a separate audio track. This is normal
when the project uses detached audio, an external BGM, or the original
sound was muted in CapCut._"

(v0.1.2 will properly detect this case via segment.volume + per-segment
coverage. Out of scope for v0.1.1.)

### Blocker 3 — sequence name fallback

Modern CapCut drafts are all literally named `draft_info.json`, so
`Path.stem` yields the useless string `"draft_info"`. Added
`_derive_timeline_name()`: when stem == "draft_info", walk up the path
skipping UUID directories and `Timelines/` to find the human-named
project dir. Now `0509.xml`, `cutsmith.xml`, `cutsmith2.xml` are
produced without needing the `-n` flag.

3 regression tests for the three layouts (legacy filename, flat
draft_info, nested under UUID).

## Decisions captured (so we don't re-debate them)

1. **Decryption of modern Jianying drafts is OUT of v0.1/v0.2 scope.**
   Reason: compliance grey area + key rotation = unstable target.
   Detect refuses cleanly. See `docs/notes/modern_jianying_75_storage.md`.

2. **Writer is a pure IR → XML transform. It does NOT synthesize audio
   tracks from `video.has_audio`.** CapCut already gives us the
   appropriate audio track when one is needed; auto-extracting on top
   doubled audio on import.

3. **Variable speed is encoded implicitly via XML duration mismatch, not
   an explicit Time Remap `<filter>`.** This is v0.1.1's biggest
   unverified assumption — needs PR import test. v0.2 may add explicit
   filter.

4. **`extra_material_refs` blanket reporting is gone.** Each ref is
   classified to a specific category; default-state refs are silently
   skipped. Trade-off: an unreferenced material isn't surfaced.

5. **Tags are not as immutable as the convention says** — for a tag
   pushed minutes ago, never publicized, and with no GitHub Release yet,
   force-moving to align with content is fine and was done once
   (17937ea → ed11bec).

6. **Embedded-audio detection (Smart Embedded Audio Fallback) deferred to
   v0.1.2.** Designed but not started. See known_limitations.md +
   user's v0.1.2 spec in conversation history.

7. **Creator validation > new features.** v0.1.2 work is held until
   creator validation feedback is in. Risk that v0.1.2 priority shifts
   based on findings (e.g., if PR doesn't auto-interpret the speed
   duration mismatch, explicit Time Remap moves up).
